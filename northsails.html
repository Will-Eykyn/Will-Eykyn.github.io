Using Machine Learning to Design Sails — North Sails

Summer 2024  ·  North Sails, Marblehead MA  ·  Data & Systems Intern
C# ML.NET LGBM SQL Server Machine Learning Data Validation IQR Outlier Reduction Correlation Analysis

Overview

This project was a summer internship at North Sails with the goal of using machine learning to automatically predict the material properties of sail battens. A batten is a stiff rod inserted into a sail to maintain its shape. The key property to predict is flexural rigidity (EI) — a measure of how stiff a batten is — which determines how the sail holds its shape in different wind conditions.

By the end of the summer I had built a working program that predicts battens for seven mainsail types: regular, full batten, high roach, boom furling (regular), square-top, boom furling (square-top), and mast furling. The output is now integrated into North Sails' internal design software and actively used by sail designers around the world.

This is a technical write-up of the approach I took, including dataset refinement, model training, and iterative tuning.

Step 1: Refine the Dataset

The goal was to predict batten width, thickness, and flexural rigidity (EI) for any mainsail Carbon/E-Glass batten. I started by building a SQL view in SQL Server Management Studio filtered to a single sail type (SailID). The view included columns: ListOrder, SailID, Area, Stiffness (EI), BattenQty, E, P, AmMax, BattenLength, LOA, Luff, FinishedFoot, and BattenStyleID.

To use luff and finished foot as reliable features, I first had to eliminate data entry errors from the database. I calculated the percentage difference between the boat's actual E/P dimensions and the sail's recorded Luff/FinishedFoot values:

-- EFoot: % difference between boat E and sail FinishedFoot ROUND(ABS(CAST(dbo.BoatGen.E AS FLOAT) - CAST(t.FinishedFoot AS FLOAT)) / CAST(dbo.BoatGen.E AS FLOAT) * 100, 2) -- PLuff: % difference between boat P and sail Luff ROUND(ABS(CAST(dbo.BoatGen.P AS FLOAT) - CAST(t.Luff AS FLOAT)) / CAST(dbo.BoatGen.P AS FLOAT) * 100, 2)

Any rows with EFoot or PLuff > 10% were filtered out as likely data entry errors.

To understand which features correlated with EI, I wrote a stored procedure in SQL Server to calculate correlation coefficients between any two columns:

EXEC sp_calculatecorrelation '[viewPath].[viewName]', 'Column1', 'Column2';

This returns a score between ±1, where values closer to ±1 indicate stronger correlation. Results near 0 indicate no meaningful relationship.

The raw data had enormous variance in EI values across different designers, even for sails of similar size. To reduce this noise, I applied IQR-based outlier reduction: I split the data into luff length groups (0–10m, 10–15m, 15–20m, 20m+) and then by List Order, calculated the interquartile range within each group, and filtered out EI values outside Q1–Q3. This kept only the "middle 50%" of values — the designs that reflect typical North Sails practice rather than outlier choices.

GIGO principle: ML models are only as good as the data you feed them. Filtering to the IQR dramatically reduced noise and caused R² values to improve significantly after retraining.

After applying the WHERE clause to enforce the IQR bounds, I re-ran the correlation procedure to verify the refined dataset showed stronger relationships between input features and EI values.

Step 2: Train the Model

I trained a separate model for every List Order (batten position) within each sail type. For most sail types, I built models for the top 6 batten positions — sails with more than 6 battens have all battens below position 6 share the LO6 prediction, since there isn't enough data to justify a 7th or 8th model.

Each model was trained in Visual Studio using the ML.NET Value Prediction scenario with these features:

  • Luff
  • FinishedFoot
  • Area
  • AmMax (adjusted modulus maximum)
  • BattenLength
  • LOA (length overall)

All features are numerical. I experimented with categorical inputs (combining SailIDs and List Orders into single models) but found the results far less accurate. The better approach was to handle categorical logic in code — routing each prediction to the correct model based on SailID and ListOrder — and keep each model focused on a single position and sail type.

I used 5-fold cross-validation and selected the Light Gradient Boosted Machine (LGBM) algorithm after testing alternatives, as it consistently produced the highest and most stable R² values. Each model was trained for 2 minutes, and the process was repeated for all List Orders within each SailID.

To convert the EI prediction into a physical batten recommendation (width and thickness), I mapped EI values to actual available batten codes. Width was determined by LOA using threshold rules derived from plotting LOA vs Width across the dataset. Thickness was initially set at the midpoint EI between adjacent available battens, then refined during tuning.

Step 3: Testing & Tuning

After training the full set of models, I wrote a C# program that takes sail parameters from the command line (sailID, luff, finishedFoot, area, amMax, loa, carbon, and batten lengths LO1–LO7), routes the inputs to the correct models, and returns a full batten prediction.

To tune the models, I selected 20 boats from the North Sails database spanning a range of sizes, brands, and types. For each boat I recorded the actual (as-built) batten EI, the raw model EI output, and the final batten prediction, then plotted all three on scatter plots per boat. Tuning involved iterating on width and thickness thresholds based on patterns I observed across the 20 boats:

  • Is the raw EI prediction or the mapped batten EI closer to the actual?
  • Is a particular batten position (e.g. LO3) consistently under-predicted?
  • Does the width jump to a larger size at too small an LOA?
After 4 iterations of tuning on a Hinckley 51 SW test case, the model's predicted EI curve closely tracked the actual as-built EI across all 6 batten positions, compared to the significant divergence seen in the initial un-tuned prediction.

I recorded every change made (what was adjusted, from what value to what value, and which boats prompted it) so that adjustments could be reversed if a change hurt performance elsewhere.

North Sails & ML in the Future

Through this project I identified several other areas where ML could reduce manual lookup work at North Sails: corner attachment ring sizing, strapping width selection, reef attachment hardware, and edge tape selection. The common thread is replacing Excel lookup tables and "check a previous project" workflows with data-driven predictions.

One future idea discussed with the team was using AI to compare a sail's original design shape to its actual flying shape, and automatically generate a proposed revision that would fly closer to the target. That is a larger-scope project requiring a full team, but it's a compelling direction for ML in sail design.

A question I left open: is ML.NET the right framework long-term? I found it well-suited to numerical regression but limited with categorical data. It may be worth exploring scikit-learn or Google's Vertex AI for future projects where categorical features are more central.

Internship Recap

I started with no clear path to solving the problem and jumped in by retraining existing models to understand the data and tools. The early approach (predicting width and thickness separately) produced batten combinations that didn't exist in practice. The key breakthrough was switching to predicting EI directly — a universal property that maps cleanly to both dimensions. Once I made that pivot, the project gained momentum quickly.

IQR-based data refinement was the other major unlock. Even Glenn Cook was surprised by the range of EI values used by North designers across similar sail sizes. Filtering to the middle 50% gave the models much cleaner signal and improved R² values substantially.

The solution is not as modular as I'd have liked — there is a lot of routing logic around the models to direct inputs to the correct SailID/ListOrder model. But this specificity is what made the predictions accurate. Once I had the template for a new SailID, adding new sail types became progressively faster.

Thanks to Shane Elliott and Glenn Cook for their guidance, patience, and for giving me real ownership of the project from day one.
